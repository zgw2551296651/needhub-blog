# Java 面试准备全攻略——从基础到 Offer

## 引言

在技术圈流传着一句话："面试造火箭，工作拧螺丝。"这句话虽然带着几分调侃，却也道出了一个不争的事实：面试所考察的知识广度和深度，往往远超日常开发工作的实际需求。你可能在项目中只需要会调用 Spring 的注解就能完成业务开发，但面试官偏偏要追问 IoC 容器的底层实现；你可能只需要写几句简单的 SQL 就能搞定数据查询，但面试官偏偏要从 B+ 树的索引结构一路问到事务隔离级别的 MVCC 实现。

面对这种现实，与其抱怨规则，不如主动适应。毕竟，面试本质上是一场筛选游戏，而"打铁还需自身硬"才是亘古不变的真理。与其在面试失败后懊恼不已，不如从现在开始，系统性地构建自己的知识体系，在面试中展现出令人信服的技术实力。

本文的定位是一份**面向中高级 Java 开发者的面试准备指南**。无论你是正在准备校招的在校学生，还是打算跳槽的社招程序员，这篇文章都会从面试官视角、知识体系构建、简历优化、投递策略、高频知识点、算法刷题以及心态管理等多个维度，为你提供一份详尽的备战路线图。我们不贩卖焦虑，只提供方法论和干货。

---

## 一、面试官的四大评估维度

在深入技术细节之前，我们先站在面试官的角度思考一个核心问题：**面试官到底在找什么样的人？** 理解评估标准，才能有的放矢地准备。

### 1.1 扎实的基础功底

基础功底是面试官最看重的维度，没有之一。原因很简单：技术框架年年变，但计算机科学的底层原理几十年如一日。一个基础扎实的人，学习新技术只是时间问题；而基础薄弱的人，即使背下了某个框架的所有 API，遇到生产环境的疑难杂症也只会手足无措。

**JVM 核心知识** 是 Java 面试的必考领域。面试官期望你不仅仅知道"JVM 分为堆、栈、方法区"这样的教科书答案，更希望你能理解：

- 内存模型的每个区域存放什么数据，为什么这样设计
- 垃圾回收的各种算法（标记-清除、复制、标记-整理、分代收集）各自解决什么问题
- 主流垃圾回收器（G1、ZGC、Shenandoah）的适用场景和设计哲学
- 类加载机制中的双亲委派模型及其打破场景

**并发编程** 则是区分初级和中高级开发者的分水岭。你需要掌握的不只是 `Thread` 和 `Runnable`，更要深入理解：

- Java 内存模型（JMM）中 happens-before 规则的含义
- `synchronized` 锁升级的完整过程（偏向锁 → 轻量级锁 → 重量级锁）
- `java.util.concurrent` 包下各工具类的实现原理和适用场景

**数据结构与算法** 几乎是所有大厂的标配考察内容。这不只是为了"刷 LeetCode"，而是为了证明你具备将现实问题抽象为计算模型的能力。

**网络与操作系统基础** 则是很多候选人容易忽视的盲区。当你被问到"从浏览器输入 URL 到页面展示，中间发生了什么"时，你是否能流畅地讲完 DNS 解析、TCP 三次握手、HTTP 请求、TLS 协商的全过程？

### 1.2 解决问题的能力

面试不仅考知识，更考能力。面试官会通过深挖你的项目经历，来判断你面对真实问题时的分析和解决能力。

**项目深挖** 通常采用 STAR 方法论（Situation、Task、Action、Result）。面试官会从一个项目切入，不断追问细节：

> "你提到优化了接口性能，具体优化了哪些环节？优化前耗时多少？优化后耗时多少？为什么选择这个方案而不是其他方案？"

这种追问方式的目的不是刁难，而是验证你是否真的深度参与了这个项目，还是只是在简历上贴了一层金。

**场景题** 也是面试中的常客。例如：

> "线上服务突然出现大量 Full GC，你会怎么排查？"
> "系统在大促期间数据库连接池耗尽，你会怎么处理？"

这类问题考察的是你的实战经验和排障思路，而不是你能否给出一个"标准答案"。

**系统设计思维** 在高级别岗位面试中尤为重要。面试官可能会让你设计一个秒杀系统、一个短链服务、或者一个分布式锁，以此来考察你对高并发、高可用、数据一致性等问题的理解深度。

### 1.3 成长潜力

对于校招和初中级岗位，面试官尤其关注候选人的成长潜力。一个学习能力强、成长速度快的人，即使当前技术栈不够熟练，也值得投入培养。

**学习方法论** 是成长潜力的重要体现。面试官可能会问："你是怎么学习一门新技术的？"或者"最近一年你学了什么新东西？"你的回答应该体现出你有系统化的学习方法，而不是遇到问题才去搜索临时方案。

**软技能** 包括沟通表达能力、团队协作能力、以及主动性。面试本身就是一个沟通场景——你的表达是否清晰有条理，能否在面试官提出问题时先理解再回答，这些都是软技能的体现。

**职业规划清晰度** 也是面试官会关注的点。当被问到"你的职业规划是什么"时，面试官想知道的是：你对自己的发展方向有没有思考？你是否打算在这个领域长期深耕？一个没有方向的候选人，往往也缺乏持续学习的动力。

### 1.4 价值观匹配

这个维度听起来比较"虚"，但在实际面试中确实存在。面试官会判断：

- 你是否认同公司的技术文化和工作方式？
- 你对加班、出差、团队规模等现实条件的期望是否与公司匹配？
- 你的职业诉求（技术深度、业务广度、管理方向）是否能在这个岗位得到满足？

价值观不匹配的候选人，即使技术能力再强，入职后的稳定性也存在风险。因此，面试是双向选择的过程——你在被评估的同时，也在评估这家公司是否适合自己。

## 二、内功心法 vs 外功招式

如果把程序员的技能体系比作武功，那么计算机基础知识就是"内功心法"，而各种技术框架和中间件则是"外功招式"。内功深厚的人，修炼任何招式都能事半功倍；而只会招式不懂内功的人，遇到变化就会手忙脚乱。

### 2.1 内功：计算机基础

**操作系统** 是程序员必须理解的基础。你需要掌握的核心概念包括：

- **进程与线程**：进程是资源分配的基本单位，线程是 CPU 调度的基本单位。理解两者的区别，以及协程（轻量级线程）的概念。
- **内存管理**：虚拟内存、页表、缺页中断、内存映射（mmap）的工作原理。
- **IO 模型**：阻塞 IO、非阻塞 IO、IO 多路复用（select/poll/epoll）、异步 IO 的区别和适用场景。理解为什么 Netty 选择了 IO 多路复用模型。

**计算机网络** 的核心知识体系：

- **TCP/IP 协议栈**：三次握手、四次挥手、滑动窗口、拥塞控制（慢启动、快重传、快恢复）。
- **HTTP/HTTPS**：HTTP/1.1 的 Keep-Alive、HTTP/2 的多路复用、HTTPS 的 TLS 握手过程。
- **Socket 编程**：Socket 的本质是操作系统提供的通信端点抽象，理解 Socket 与 TCP 的关系。

**数据结构** 是一切算法的基石：

- **数组与链表**：最基本的线性结构，理解它们在内存中的布局差异以及对随机访问和插入删除操作的时间复杂度影响。
- **树**：二叉树、平衡二叉树（AVL、红黑树）、B 树与 B+ 树。理解为什么 MySQL 的 InnoDB 引擎选择 B+ 树作为索引结构。
- **图**：有向图、无向图、拓扑排序、最短路径算法。在依赖分析、任务调度等场景中有实际应用。
- **哈希**：哈希函数的设计、哈希冲突的处理（开放寻址法、链地址法），以及一致性哈希在分布式系统中的应用。

**算法** 的核心思想：

- **排序**：快速排序、归并排序、堆排序的原理和时间复杂度分析。
- **搜索**：二分查找、深度优先搜索（DFS）、广度优先搜索（BFS）。
- **动态规划**：理解状态定义和状态转移方程的本质——将大问题拆解为重叠子问题。
- **贪心**：每一步选择局部最优解，适用于具有贪心选择性质的问题。

### 2.2 外功：市场技术栈

**Java 生态** 是大多数 Java 开发者的核心技能：

- **Spring 全家桶**：Spring Framework、Spring Boot、Spring Cloud 是必须掌握的。理解 Spring 的 IoC 和 AOP 原理，而不仅仅是会用注解。Spring Boot 的自动配置原理（`@EnableAutoConfiguration` 的加载机制）也是高频考点。
- **MyBatis / MyBatis-Plus**：理解 MyBatis 的缓存机制（一级缓存、二级缓存）、插件机制、以及动态 SQL 的实现原理。

**中间件** 是中高级面试的重点领域：

- **Redis**：五种基本数据类型的底层实现、持久化策略（RDB/AOF）、主从复制、哨兵模式、Cluster 模式、缓存穿透/击穿/雪崩的解决方案。
- **消息队列**（Kafka/RocketMQ）：消息模型、顺序消息、消息可靠性、消息积压处理、高可用架构设计。
- **MySQL 调优**：索引优化、慢查询分析、执行计划（EXPLAIN）解读、分库分表策略、主从复制与读写分离。

**微服务架构** 是现代分布式系统的主流范式：

- **Dubbo / Spring Cloud**：RPC 框架的核心原理（序列化、网络通信、服务注册与发现）。
- **注册中心**：Nacos、Consul、ZooKeeper 的对比和 CP/AP 选型。
- **网关**：Spring Cloud Gateway 的过滤器链机制、限流策略。
- **服务治理**：熔断降级（Hystrix/Sentinel）、链路追踪（SkyWalking/Zipkin）。

**云原生** 已经成为趋势：

- **Docker**：容器化的核心概念（镜像、容器、仓库）、Dockerfile 编写最佳实践、容器网络模型。
- **Kubernetes（K8s）**：Pod、Deployment、Service、ConfigMap 等核心资源对象的概念，以及基本的集群管理操作。

## 三、简历优化指南

简历是你给面试官的第一印象，也是面试话题的"导航图"。一份好的简历，能引导面试官问你擅长的问题；一份差的简历，则可能让你连面试机会都拿不到。

### 3.1 个人信息

个人信息部分应该简洁明了，突出与技术相关的关键信息：

- **必填项**：姓名、联系方式（手机号/邮箱）、GitHub 地址（如果有高质量项目）、技术博客地址（如果有持续输出）。
- **可选填**：年龄、学历、工作年限。
- **避免填写**：身高体重、血型星座、兴趣爱好（除非与技术强相关，如"开源贡献"）。

### 3.2 项目经验

项目经验是简历的核心部分，也是面试官最关注的内容。推荐使用 **STAR 方法论** 来组织项目描述：

**S（Situation，背景）**：项目所处的业务背景和技术环境是什么？

> 例：某电商平台在大促期间面临订单系统吞吐量不足的问题，峰值 QPS 达到 5 万+，原有单体架构无法满足性能要求。

**T（Task，任务）**：你在项目中承担的具体职责是什么？

> 例：作为核心开发成员，负责订单服务的微服务拆分和性能优化工作。

**A（Action，行动）**：你采取了哪些具体的技术方案和行动？

> 例：将订单服务拆分为订单创建、支付、物流三个独立微服务；引入 Redis 做预扣库存，使用 RocketMQ 做异步削峰；对热点数据采用本地缓存 + 分布式缓存的两级缓存策略。

**R（Result，结果）**：最终取得了什么成果？**务必量化。**

> 例：优化后订单接口 P99 延迟从 800ms 降至 120ms，系统峰值 QPS 从 2000 提升至 12000，大促期间零故障。

**量化成果** 是项目描述中最有说服力的部分。以下是一些量化思路：

- 性能指标：QPS、TPS、延迟（P50/P99）、吞吐量
- 可靠性指标：可用性（99.99%）、故障率、MTTR（平均恢复时间）
- 效率指标：构建时间缩短比例、部署频率提升比例
- 业务指标：用户转化率提升、成本降低金额

### 3.3 技能描述

技能描述部分是面试官快速评估你技术栈匹配度的依据，需要注意以下几点：

**技术名词正确书写** 是基本素养，也是很多候选人忽视的细节：

- 正确：`Java`、`MySQL`、`Redis`、`Spring Boot`、`MyBatis`、`Kubernetes`、`Docker`
- 错误：`java`、`mysql`、`redis`、`springboot`、`mybatis`、`k8s`（简历中首次出现应写全称）

**熟练度分级** 要实事求是，因为面试官会根据你的熟练度来调整问题深度：

- **精通**：深入理解底层原理，能进行源码级别的分析和调优，在复杂场景下能灵活运用。**慎用——写了"精通"就要准备好被深挖。**
- **熟悉**：有丰富的实际使用经验，理解核心原理，能独立解决常见问题和疑难问题。
- **掌握**：有实际项目使用经验，能完成日常开发任务，了解基本原理。
- **了解**：知道这项技术是什么，解决什么问题，但没有深入使用过。

**避免模糊描述**。不要写"熟悉各种设计模式"——你熟悉哪些设计模式？在什么场景下用过？改为"熟悉常用设计模式（工厂、策略、观察者、模板方法），并在实际项目中多次应用"会更有说服力。

## 四、投递时机与渠道

"好风凭借力，送我上青云。"选择合适的投递时机和渠道，能显著提高你的面试成功率和 Offer 质量。

### 4.1 校招时间线

校招是应届生进入大厂的最佳通道，时间节奏非常关键：

- **秋招（8 月至 11 月）**：这是校招的主力战场，岗位数量最多、质量最好。大多数大厂会在 7-8 月开放提前批，8-10 月集中笔试面试，11 月陆续发放 Offer。建议从暑假前（6 月）就开始准备。
- **春招（2 月至 5 月）**：春招本质上是秋招的补录，岗位数量和质量都不如秋招，但竞争压力也相对较小。适合秋招失利的同学把握最后机会。
- **提前批和实习转正**：很多大厂会开设提前批通道（通常 6-7 月），提前批的优势是可以提前锁定 Offer，即使失败也不影响正式批。此外，通过暑期实习转正也是进入大厂的优质通道——实习转正的通过率通常远高于校招。

### 4.2 社招黄金期

社招的时间窗口相对灵活，但也有规律可循：

- **金三银四（3-4 月）**：春节后是传统的跳槽旺季，年终奖已到手，新财年的 HC（人员编制）刚刚释放，市场上的岗位数量充裕。
- **金九银十（9-10 月）**：下半年另一个跳槽窗口，部分公司会在年底前做最后一轮人员补充。

**提前 2-3 个月准备** 是社招的基本策略。如果你计划在 3 月投递，那么从 1 月就应该开始系统性地复习和刷题。不要等到决定离职了才开始准备——仓促上阵的结果往往是"跳了个寂寞"。

### 4.3 投递渠道

不同的投递渠道，效率和效果差异巨大：

- **官网直投**：最稳妥的方式，但效率较低，适合目标明确、投递数量不大的情况。
- **招聘平台**（Boss 直聘、拉勾、猎聘等）：覆盖面广，适合批量投递和了解市场行情。注意：在平台上要主动出击，积极与 HR 沟通。
- **内推**：这是最高效的渠道。通过内部员工推荐，简历通常能跳过初筛直接进入面试官手中。内推可以通过前同事、校友群、技术社区等方式获取。但要注意：内推人的背书也很重要——如果你的简历质量太差，内推人也会犹豫。
- **技术社区与猎头**：在掘金、GitHub、Stack Overflow 等社区活跃，有时候会有机会主动找上门。猎头渠道适合有 3 年以上经验的开发者，好的猎头能帮你匹配更合适的岗位和争取更好的薪资。

## 五、面试高频知识点清单

以下是 Java 面试中最高频的知识点清单，建议逐一攻克，做到能用自己的语言清晰表述。

### 5.1 Java 基础

**HashMap 原理** 几乎是必考题。你需要讲清楚：

- JDK 1.8 中 HashMap 的底层结构是"数组 + 链表 + 红黑树"
- `put` 操作的完整流程：计算 hash → 定位桶位置 → 判断是否扩容 → 插入/更新节点
- 链表转红黑树的条件（链表长度 ≥ 8 且数组长度 ≥ 64）
- 扩容机制（2 倍扩容，尾插法解决 JDK 1.7 的环形链表问题）

```java
// HashMap 的核心参数
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4; // 16
static final int MAXIMUM_CAPACITY = 1 << 30;
static final float DEFAULT_LOAD_FACTOR = 0.75f;
static final int TREEIFY_THRESHOLD = 8;      // 链表转红黑树阈值
static final int UNTREEIFY_THRESHOLD = 6;    // 红黑树退化为链表阈值
static final int MIN_TREEIFY_CAPACITY = 64;  // 树化的最小数组长度
```

**ConcurrentHashMap** 的演进过程也要掌握：

- JDK 1.7：分段锁（Segment），每个 Segment 继承 ReentrantLock
- JDK 1.8：CAS + synchronized，锁粒度细化到每个桶（Node），并发度大幅提升

**String / StringBuilder / StringBuffer** 的区别：

- `String` 是不可变对象，底层是 `final char[]`（JDK 9+ 改为 `byte[]`）
- `StringBuilder` 是可变对象，线程不安全，性能最好
- `StringBuffer` 是可变对象，线程安全（方法用 `synchronized` 修饰），性能较差

**泛型擦除** 是另一个高频话题：Java 泛型是编译期检查，运行时会被擦除为 `Object`（或上界类型）。理解通配符 `? extends T`（上界，只读）和 `? super T`（下界，只写）的使用场景，以及 PECS 原则。

### 5.2 JVM

**内存区域划分**：

- **堆**：存放对象实例，是 GC 的主战场。分为新生代（Eden + Survivor）和老年代。
- **方法区（元空间）**：存放类信息、常量池、静态变量。JDK 8 之后由元空间（Metaspace）替代永久代，使用本地内存。
- **虚拟机栈**：每个线程私有，存放栈帧（局部变量表、操作数栈、动态链接、方法出口）。
- **本地方法栈**：为 Native 方法服务。
- **程序计数器**：线程私有，记录当前线程执行的字节码行号。

**GC 算法与垃圾回收器**：

```
// 分代收集策略
新生代 → 复制算法（Minor GC，存活率低，复制成本低）
老年代 → 标记-清除 或 标记-整理（Major GC / Full GC）

// 主流垃圾回收器对比
Parallel Scavenge   → 吞吐量优先
CMS                 → 低延迟，但会产生内存碎片
G1（Garbage First） → 兼顾吞吐和延迟，JDK 9+ 默认
ZGC                 → 超低延迟（< 10ms），JDK 15 正式可用
Shenandoah          → Red Hat 开发，与 ZGC 类似的低延迟目标
```

**类加载机制——双亲委派模型**：

```java
// 双亲委派的核心逻辑
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            // 2. 委托父加载器加载
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                c = findBootstrapClassOrNull(name);
            }
            // 3. 父加载器无法加载，自己加载
            if (c == null) {
                c = findClass(name);
            }
        }
        return c;
    }
}
```

打破双亲委派的典型场景：Tomcat 的 WebAppClassLoader（子委派）、JDBC 的 SPI 机制（线程上下文类加载器）、OSGi 的模块化热部署。

**JVM 调优常用参数**：

```bash
# 堆内存设置
-Xms4g -Xmx4g          # 设置堆的初始值和最大值（建议设置为相同值）
-Xmn2g                  # 设置新生代大小

# GC 选择与日志
-XX:+UseG1GC            # 启用 G1 垃圾回收器
-XX:MaxGCPauseMillis=200 # G1 目标最大停顿时间
-XX:+PrintGCDetails     # 打印 GC 详细信息
-Xloggc:/var/log/gc.log # GC 日志输出路径

# OOM 排查
-XX:+HeapDumpOnOutOfMemoryError           # OOM 时自动导出堆快照
-XX:HeapDumpPath=/var/log/heapdump.hprof  # 堆快照保存路径
```

### 5.3 并发编程

**synchronized vs ReentrantLock**：

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 锁的实现 | JVM 层面（Monitor） | API 层面（AQS） |
| 可中断 | 不支持 | 支持（`lockInterruptibly`） |
| 公平锁 | 非公平 | 支持公平/非公平 |
| 条件变量 | 单一 `wait/notify` | 多个 `Condition` |
| 自动释放 | 是（退出代码块） | 否（需手动 `unlock`） |

**volatile 语义**：保证可见性和有序性，但不保证原子性。实现原理是通过内存屏障（Memory Barrier）禁止指令重排序，并在写操作时强制刷新主内存、读操作时强制从主内存读取。典型应用场景是双重检查锁定（DCL）单例模式中的 `instance` 变量。

**AQS（AbstractQueuedSynchronizer）** 是 JUC 包的基础框架：

```java
// AQS 的核心结构
// 1. 一个 volatile int state 变量，表示同步状态
// 2. 一个 FIFO 的双向链表（CLH 队列变体），存放等待线程节点
// 3. 通过 CAS 操作修改 state，失败的线程进入队列等待

// ReentrantLock 基于 AQS 实现：
// - state = 0 表示锁可用
// - state > 0 表示锁被持有，数值代表重入次数
// - 获取锁：CAS(state, 0, 1)，失败则入队
// - 释放锁：state - 1，减为 0 时唤醒队列中下一个节点
```

**线程池核心参数**（`ThreadPoolExecutor`）：

```java
public ThreadPoolExecutor(
    int corePoolSize,       // 核心线程数（即使空闲也不回收）
    int maximumPoolSize,    // 最大线程数
    long keepAliveTime,     // 非核心线程空闲存活时间
    TimeUnit unit,          // 时间单位
    BlockingQueue<Runnable> workQueue,    // 任务队列
    ThreadFactory threadFactory,          // 线程工厂
    RejectedExecutionHandler handler      // 拒绝策略
)
```

任务提交流程：核心线程未满 → 创建核心线程 → 核心线程满 → 队列未满 → 放入队列 → 队列满 → 最大线程未满 → 创建非核心线程 → 最大线程满 → 执行拒绝策略。

四种内置拒绝策略：`AbortPolicy`（抛异常，默认）、`CallerRunsPolicy`（调用者线程执行）、`DiscardPolicy`（静默丢弃）、`DiscardOldestPolicy`（丢弃最旧任务）。

### 5.4 Spring

**IOC（控制反转）** 的核心思想是将对象的创建和管理权交给 Spring 容器，而不是由开发者手动 `new`。实现方式是依赖注入（DI），包括构造器注入、Setter 注入和字段注入（`@Autowired`）。

**AOP（面向切面编程）** 的实现原理：

- 如果目标对象实现了接口 → 使用 JDK 动态代理（基于 `Proxy` 和 `InvocationHandler`）
- 如果目标对象没有实现接口 → 使用 CGLIB（通过生成目标类的子类来实现代理）
- Spring Boot 2.x 默认使用 CGLIB 代理

**Bean 生命周期** 是经典面试题，完整流程如下：

1. 实例化（Instantiation）：通过反射创建对象
2. 属性赋值（Populate）：注入依赖（`@Autowired`、`@Value`）
3. 初始化（Initialization）：
   - 调用 `BeanNameAware`、`BeanFactoryAware` 等感知接口
   - 调用 `BeanPostProcessor#postProcessBeforeInitialization`
   - 调用 `@PostConstruct`、`InitializingBean#afterPropertiesSet`、`init-method`
   - 调用 `BeanPostProcessor#postProcessAfterInitialization`（AOP 代理在此生成）
4. 使用（In Use）
5. 销毁（Destruction）：调用 `@PreDestroy`、`DisposableBean#destroy`、`destroy-method`

**事务传播机制** 定义了方法之间调用时事务的行为：

- `REQUIRED`（默认）：当前有事务则加入，无则新建
- `REQUIRES_NEW`：无论当前有无事务，都新建一个独立事务
- `NESTED`：当前有事务则创建嵌套事务（Savepoint），无则新建
- `SUPPORTS`：当前有事务则加入，无则非事务执行
- `NOT_SUPPORTED`、`MANDATORY`、`NEVER`：较少使用

**循环依赖** 的解决方案——三级缓存：

```
一级缓存 singletonObjects      → 存放完全初始化好的 Bean
二级缓存 earlySingletonObjects → 存放提前暴露的 Bean（已实例化但未完成属性注入）
三级缓存 singletonFactories    → 存放 Bean 的 ObjectFactory（用于生成早期引用）
```

当 A 依赖 B、B 依赖 A 时：A 实例化 → 将 A 的工厂放入三级缓存 → A 注入属性发现需要 B → B 实例化 → B 注入属性发现需要 A → 从三级缓存获取 A 的早期引用 → B 完成初始化 → A 完成初始化。注意：构造器注入的循环依赖无法解决，因为对象还没有被创建出来。

## 六、算法刷题策略

算法是面试中很多人最头疼的环节，但也是最有章可循的部分。

**高频题型分布** 大致如下：

- 数组/字符串：约 20%（双指针、滑动窗口）
- 链表：约 10%（反转、合并、环检测）
- 树：约 15%（遍历、路径问题、LCA）
- 动态规划：约 20%（背包、区间 DP、状态压缩）
- 栈/队列/堆：约 10%（单调栈、优先队列）
- 图论：约 10%（BFS/DFS、拓扑排序）
- 其他：约 15%（位运算、数学、设计题）

**LeetCode 刷题路线建议**：

1. **入门阶段**（2-3 周）：从 Easy 题开始，熟悉各种基本数据结构和算法模板。推荐"LeetCode Hot 100"中的 Easy 题。
2. **进阶阶段**（4-6 周）：集中攻克 Medium 题，按专题刷（比如先刷完所有二叉树相关的 Medium 题，再刷动态规划）。推荐"剑指 Offer"系列。
3. **冲刺阶段**（2-3 周）：做 Hard 题中的高频题，进行限时模拟练习。推荐"LeetCode Hot 100"中的 Hard 题和各公司高频题库。

**面试中算法题的应对技巧**：

- **先沟通再写代码**：拿到题目后，不要急着写代码。先和面试官确认题意，讨论可能的解法和时间复杂度，确认思路正确后再动手。
- **从暴力解法开始**：如果一时想不到最优解，先给出暴力解法，然后再逐步优化。面试官看重的是你的思考过程，而不只是最终答案。
- **注意边界条件**：空数组、单元素、负数、极大值等边界情况往往是 Bug 的温灾区。
- **练习白板编程**：面试中可能需要在白板上或者没有代码提示的编辑器中写代码，平时刷题要有意识地练习脱离 IDE 编程的能力。

```java
// 示例：经典的双指针法——两数之和 II（有序数组）
public int[] twoSum(int[] numbers, int target) {
    int left = 0, right = numbers.length - 1;
    while (left < right) {
        int sum = numbers[left] + numbers[right];
        if (sum == target) {
            return new int[]{left + 1, right + 1};
        } else if (sum < target) {
            left++;   // 和太小，左指针右移
        } else {
            right--;  // 和太大，右指针左移
        }
    }
    return new int[]{-1, -1};
}
```

## 七、模拟面试与心态管理

**模拟面试的重要性** 怎么强调都不过分。很多人的知识储备其实已经足够了，但在真实面试场景中因为紧张、表达不清、时间把控不好等原因而发挥失常。模拟面试可以帮助你：

- 适应面试的对话节奏，学会在被追问时保持冷静和条理
- 发现自己知识体系中的盲点——你以为自己懂了，但讲出来的时候才发现其实理解得不够深入
- 练习时间管理——算法题通常只有 20-30 分钟，你需要在有限时间内完成思考、编码和测试

找模拟面试的渠道：朋友或同事互相面试、牛客网的模拟面试功能、Pramp（英文面试平台）、或者自己对着镜子/录音练习自我介绍和项目讲解。

**常见陷阱和应对**：

- **"你有什么想问我的？"**：这绝不是走个过场。准备 2-3 个有深度的问题，比如"团队目前最大的技术挑战是什么？""这个岗位未来的发展方向是什么？"避免问"加班多不多""年假有多少天"这类问题——这些可以拿到 Offer 后再问 HR。
- **"你对加班怎么看？"**：面试官想知道的是你的工作态度和灵活性。诚实回答即可，但避免给出极端的回答（"绝对不加班"或"随时都可以加班"都不太合适）。
- **"你的期望薪资是多少？"**：提前做好市场调研（可以参考 Levels.fyi、脉脉薪资爆料等），给出一个合理范围，而不是一个具体数字。表达出"我更看重发展机会，薪资方面希望能在市场合理范围内协商"的态度。
- **"说说你的缺点"**：不要说"我太追求完美了"这种明显的伪装。选一个真实但不致命的缺点，并说明你正在如何改进。

**"准备能消除未知"**——这是面试心态管理的核心原则。面试焦虑的本质是对未知的恐惧：不知道会被问什么，不知道自己能不能答上来，不知道结果会怎样。而充分的准备能最大限度地减少不确定性——当你已经系统地复习了所有高频知识点，做了足够多的模拟面试，刷了 200+ 道 LeetCode 题之后，你会发现面试不再是一件令人恐惧的事情，而是一次展示自己能力的机会。

记住：**面试不是一场考试，而是一次对话。** 面试官不是你的敌人，他也在寻找合适的人选。把面试当作一次和同事讨论技术问题的机会，保持自信和从容，你的表现会好得多。

## 总结

面试准备是一场持久战，需要系统性的规划和持续性的投入。以下是几条核心建议：

**制定个人复习计划**：根据自己的实际情况，将复习内容分配到每天的时间表中。建议按照"基础 → 框架 → 项目 → 算法"的顺序推进，每天保证 2-3 小时的有效学习时间。如果你是在职准备跳槽，可以利用早起和晚上的碎片时间。

**重视输出而非只是输入**：看书、看视频、看博客都是"输入"，而真正能检验你是否掌握的只有"输出"——你能否用自己的语言把一个知识点讲清楚？你能否不看资料手写出一道算法题？尝试写技术博客、给别人讲解知识点，这些"输出"行为会极大加深你的理解。

**持续学习才是终极竞争力**：面试准备不应该是一次性的突击行为，而应该融入你的日常工作习惯中。保持对新技术的好奇，定期阅读优秀的开源代码，在实践中不断总结和反思——这样，当下一次面试机会来临时，你只需要做一些针对性的准备，而不是从零开始恶补。

最后，借用一句话与各位共勉：**机会总是留给有准备的人。** 祝你在面试中展现出最好的自己，拿到心仪的 Offer。
