# Redis 持久化、集群架构与高可用方案

> 本文系统梳理 Redis 从单机持久化到分布式集群的完整技术栈，涵盖 RDB/AOF/混合持久化、线程模型、主从/哨兵/Cluster 三种集群架构、Hash Slot 路由、脑裂防御、内存淘汰策略等核心知识。适合架构设计与面试备考。

## 引言

Redis 作为内存数据库，数据全部存储在 RAM 中。这意味着一旦进程退出或机器断电，数据就会丢失。在生产环境中，数据丢失可能意味着用户会话中断、订单数据丢失、排行榜数据清零等严重后果。因此，持久化机制解决的是**数据安全**问题——如何将内存数据可靠地落盘，在进程重启后快速恢复。

然而，持久化只是单机层面的保障。当业务规模增长，单机 Redis 面临三大挑战：内存容量上限（通常 64GB 左右）、吞吐量瓶颈（单线程约 10w QPS）、单点故障风险。集群架构解决的是**高可用与水平扩展**问题——如何在节点故障时自动切换，如何突破单机内存和吞吐量上限，如何在多节点间均匀分配数据。

从单机持久化到主从复制，从哨兵监控到 Redis Cluster 自动分片，Redis 的高可用方案经历了多次演进。每种方案都有其适用场景和局限性，理解它们的原理和权衡是架构师的核心能力之一。本文将从持久化到集群，由浅入深地拆解 Redis 的生产级架构设计。

```
  Redis 生产架构全景
  ┌──────────────────────────────────────────────────────┐
  │                                                      │
  │   持久化层           集群层              内存管理层     │
  │   ────────          ──────              ──────────    │
  │   RDB快照           主从复制             TTL管理       │
  │   AOF日志           哨兵模式             惰性删除      │
  │   混合持久化         Redis Cluster        定期采样      │
  │                     Hash Slot            淘汰策略      │
  │                     脑裂防御                           │
  └──────────────────────────────────────────────────────┘
```

## 一、RDB 快照持久化

### 1.1 基本原理

RDB（Redis Database）是 Redis 的默认持久化方式，将某一时刻的全量内存数据以二进制快照的形式写入磁盘文件（默认 `dump.rdb`）。

```bash
# 手动触发
SAVE           # 阻塞主线程，生产环境禁用
BGSAVE         # fork 子进程，主线程继续服务

# 查看最后一次 RDB 保存时间
LASTSAVE
```

### 1.2 自动触发配置

```conf
# redis.conf
save 900 1      # 900秒内至少有1次写操作
save 300 10     # 300秒内至少有10次写操作
save 60 10000   # 60秒内至少有10000次写操作

# 任一条件满足即触发 BGSAVE
# 设为 save "" 可完全禁用 RDB
```

### 1.3 SAVE vs BGSAVE

```
  SAVE（阻塞式）                    BGSAVE（非阻塞式）
  ─────────────                    ─────────────────
  主线程直接执行 rdbSave()          主线程 fork() 子进程
  ┌──────────┐                    ┌──────────┐
  │ 主线程    │                    │ 主线程    │──→ 继续处理命令
  │ 执行     │                    │          │
  │ rdbSave  │                    └──────────┘
  │ (阻塞!)  │                    ┌──────────┐
  │          │                    │ 子进程    │
  └──────────┘                    │ 执行     │
                                  │ rdbSave  │
  期间所有客户端                   │ (独立)   │
  请求被阻塞                      └──────────┘
                                  利用 COW 机制
                                  共享父进程内存页

  COW（Copy-On-Write）原理：
  1. fork 后子进程共享父进程的所有内存页（虚拟内存映射）
  2. 父进程收到写命令时，OS 内核将该内存页复制一份
  3. 父进程修改副本，子进程继续读原页面
  4. 子进程 rdbSave 遍历的是 fork 时刻的数据快照
```

### 1.4 RDB 优缺点

| 优点 | 缺点 |
|------|------|
| 文件紧凑，适合备份和灾难恢复 | 两次快照之间的数据可能丢失 |
| 恢复速度极快（直接加载二进制） | fork 大内存进程时可能阻塞数百毫秒 |
| 对主线程性能影响小（子进程处理） | COW 可能导致物理内存翻倍 |

## 二、AOF 追加日志持久化

### 2.1 基本原理

AOF（Append Only File）记录每一条写命令，以 Redis 协议格式追加到日志文件末尾。恢复时重放所有命令即可重建数据。

```conf
# redis.conf
appendonly yes                    # 开启 AOF
appendfilename "appendonly.aof"   # AOF 文件名
```

```
  AOF 文件内容示例（Redis 协议格式）
  ┌─────────────────────────────────────┐
  │ *2\r\n                              │
  │ $6\r\n                              │
  │ SELECT\r\n                          │
  │ $1\r\n                              │
  │ 0\r\n                               │
  │ *3\r\n                              │
  │ $3\r\n                              │
  │ SET\r\n                             │
  │ $4\r\n                              │
  │ name\r\n                            │
  │ $7\r\n                              │
  │ NeedHub\r\n                         │
  └─────────────────────────────────────┘
  每条命令都是标准 RESP 协议
  恢复时逐条解析执行即可
```

### 2.2 三种 fsync 模式

```conf
# always：每条命令执行后立即 fsync
appendfsync always

# everysec：每秒 fsync 一次（推荐，性能与安全平衡）
appendfsync everysec

# no：由操作系统决定何时 fsync
appendfsync no
```

```
  三种 fsync 模式对比
  ┌────────────────────────────────────────────────────────┐
  │ 模式       数据安全性      性能         适用场景         │
  │ ────       ──────────     ────         ────────        │
  │ always     最高(最多丢1条) 最低          金融级数据      │
  │                          ~数百TPS                     │
  │                                                       │
  │ everysec   较高(最多丢1秒) 高            绝大多数场景    │
  │                          ~10w+ TPS                    │
  │                                                       │
  │ no         最低(OS决定)    最高          可容忍数据丢失  │
  └────────────────────────────────────────────────────────┘
```

### 2.3 AOF 重写（Rewrite）

随着时间推移，AOF 文件会越来越大（同一个 key 可能被 SET 多次）。AOF 重写通过生成当前数据集的最小命令集来压缩文件。在生产环境中，AOF 文件膨胀是一个需要密切监控的指标——过大的 AOF 文件不仅占用磁盘空间，更严重的是会显著延长 Redis 重启时的数据恢复时间。一个 10GB 的 AOF 文件可能需要数分钟才能重放完成，期间 Redis 无法提供任何服务。

AOF 重写的另一个需要注意的点是重写期间的内存开销。重写过程需要 fork 子进程，同时主线程的新命令需要缓存在 AOF 重写缓冲区中。如果重写期间写入量很大，重写缓冲区可能消耗大量内存。生产环境中建议将 `auto-aof-rewrite-percentage` 设为 100，`auto-aof-rewrite-min-size` 根据数据量合理设置，避免过于频繁的重写操作。

```conf
# 自动重写触发条件
auto-aof-rewrite-percentage 100   # AOF 文件比上次重写后增大 100%
auto-aof-rewrite-min-size 64mb    # 且至少 64MB

# 手动触发
BGREWRITEAOF
```

```
  AOF 重写过程
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  重写前 AOF：                重写后 AOF：          │
  │  SET x 1                     SET x 3             │
  │  SET x 2                     SET y "hello"       │
  │  SET x 3                     HSET obj f1 v1      │
  │  SET y "hello"               ZADD board 100 "a"  │
  │  HSET obj f1 v1                                  │
  │  INCR x  ← 已合并到 SET x 3                      │
  │  ZADD board 100 "a"                              │
  │  DEL z  ← 已删除的 key 不再出现                    │
  │  SET z 1                                         │
  │  DEL z                                           │
  │                                                  │
  │  9 条命令 → 4 条命令                               │
  └──────────────────────────────────────────────────┘

  重写过程（BGREWRITEAOF）：
  1. fork 子进程
  2. 子进程遍历内存，生成最小命令集写入临时文件
  3. 重写期间主线程的新命令同时追加到 AOF 重写缓冲区
  4. 子进程完成后，将重写缓冲区内容追加到临时文件末尾
  5. 原子 rename 替换旧 AOF 文件
```

## 三、混合持久化（Redis 4.0+）

### 3.1 设计理念

在 Redis 4.0 之前，RDB 和 AOF 是两种互斥的选择：RDB 恢复快但可能丢数据，AOF 数据安全但恢复慢。很多团队被迫同时开启两种持久化，既用 RDB 做定期备份，又用 AOF 保证数据安全，这带来了额外的磁盘 IO 和内存开销。

混合持久化的出现完美解决了这个两难问题。它结合了 RDB 的快速恢复和 AOF 的数据安全优势：AOF 重写时，将当前全量数据以 RDB 格式写入文件头部，重写期间的新增命令以 AOF 格式追加到文件尾部。恢复时先加载 RDB 快照（极快），再执行增量 AOF 命令（通常很少），兼顾了恢复速度和数据完整性。

```conf
# redis.conf
aof-use-rdb-preamble yes    # 开启混合持久化（4.0+ 默认开启）
```

```
  混合持久化文件格式
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │  ┌──────────────────┐  ┌──────────────────────┐ │
  │  │   RDB 二进制快照   │  │  AOF 增量命令（RESP） │ │
  │  │  (重写时刻全量数据) │  │ (重写到完成期间的新   │ │
  │  │                  │  │  增/改/删命令)         │ │
  │  └──────────────────┘  └──────────────────────┘ │
  │         ↑                       ↑               │
  │    恢复速度快              数据完整性保证          │
  │                                                 │
  │  文件头：REDIS 魔数 → RDB 解析器                  │
  │  遇到 RDB 末尾后 → 切换为 AOF 解析器继续执行       │
  └─────────────────────────────────────────────────┘
```

### 3.2 恢复流程

1. 检测 AOF 文件头，判断是否为 RDB 格式（以 `REDIS` 魔数开头）
2. 如果是混合格式，先用 RDB 加载器加载快照部分
3. 再用 AOF 加载器执行增量命令部分
4. 数据恢复完成

## 四、数据安全的真相

### 4.1 fsync 不等于绝对安全

即使使用 `appendfsync always`，数据仍然有丢失风险：

```
  写命令的完整链路

  Redis 主线程
      │
      ▼
  write()  →  OS 内核页缓存(Page Cache)
      │              │
      │              ▼
      │         磁盘控制器缓存
      │              │
      │              ▼
      └──── fsync() → 磁盘介质

  风险点：
  1. fsync 返回成功，数据可能还在磁盘控制器缓存中
  2. 磁盘控制器断电 → 缓存数据丢失
  3. SSD 的 FTL（Flash Translation Layer）可能延迟写入
  4. 文件系统本身的元数据更新可能不同步
```

**生产建议**：
- 对于绝大多数业务，`everysec` 已经足够安全
- 真正需要零丢失，应使用主从同步 + 应用层幂等 + 对账机制
- 定期备份 RDB 文件到远程存储（S3/OSS），作为最后的安全网

## 五、Redis 线程模型

### 5.1 为什么是单线程

Redis 核心操作使用单线程模型，这是一个**刻意为之**的设计决策：

```
  Redis 为什么不用多线程？
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  1. 纯内存操作，瓶颈在网络 IO 而非 CPU             │
  │     - 单次 GET/SET 耗时 < 1微秒                   │
  │     - 单线程已能处理 10w+ QPS                     │
  │                                                  │
  │  2. 避免并发控制的复杂性                           │
  │     - 无需锁，无死锁风险                          │
  │     - 无需处理竞态条件                            │
  │     - 所有数据结构操作天然是原子的                  │
  │                                                  │
  │  3. IO 多路复用已足够高效                         │
  │     - epoll/kqueue 单线程管理数万连接              │
  │     - 避免了线程切换和锁竞争的开销                  │
  └──────────────────────────────────────────────────┘
```

### 5.2 IO 多路复用模型

```
  Redis IO 多路复用架构
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  Client1 ─┐                                      │
  │  Client2 ──┤    ┌────────────┐    ┌──────────┐  │
  │  Client3 ──┼──→ │ epoll_wait │──→ │ 事件分发器 │  │
  │  ...      ──┤    │ (阻塞等待) │    │          │  │
  │  ClientN ─┘    └────────────┘    └────┬─────┘  │
  │                                       │         │
  │                    ┌──────────────────┼──────┐  │
  │                    ▼                  ▼      ▼  │
  │              可读事件              可写事件   连接事件 │
  │           (读命令,解析)          (回响应)  (accept)│
  │                    │                              │
  │                    ▼                              │
  │           ┌──────────────┐                       │
  │           │ 命令执行引擎   │                       │
  │           │ (单线程串行)   │                       │
  │           └──────────────┘                       │
  │                    │                              │
  │                    ▼                              │
  │           ┌──────────────┐                       │
  │           │ 写回响应缓冲区 │                       │

  └──────────────────────────────────────────────────┘
```

## 六、三种集群架构

### 6.1 主从复制（Master-Slave）

```
  主从复制架构
  ┌─────────┐
  │  Master  │ ←── 客户端写入
  │ (读写)   │
  └────┬─────┘
       │ 异步复制
       ├──→ ┌─────────┐
       │    │ Slave 1  │ ←── 客户端读取
       │    └─────────┘
       └──→ ┌─────────┐
            │ Slave 2  │ ←── 客户端读取
            └─────────┘

  特点：
  - 手动配置，无自动故障转移
  - Master 挂了需要人工切换
  - 适合读多写少、对可用性要求不高的场景
```

```bash
# 配置从节点
replicaof master_host master_port

# 查看复制状态
INFO replication
```

**全量同步 vs 增量同步**：
- 首次连接或断线过久：全量同步（Master 执行 BGSAVE，发送 RDB 文件 + 缓冲区增量）
- 短暂断线重连：增量同步（基于 offset，Master 发送积压缓冲区中的差异数据）

```conf
# 复制积压缓冲区大小（默认 1MB，大流量建议调大）
repl-backlog-size 256mb
```

### 6.2 哨兵模式（Sentinel）

```
  哨兵模式架构
  ┌─────────┐    ┌─────────┐    ┌─────────┐
  │Sentinel1│    │Sentinel2│    │Sentinel3│   ← 哨兵集群(奇数个)
  └────┬────┘    └────┬────┘    └────┬────┘
       │               │               │
       └───────────────┼───────────────┘
                       │ 监控 + 投票
                       ▼
  ┌─────────────────────────────────────┐
  │           Redis 主从集群              │
  │                                     │
  │  ┌─────────┐                        │
  │  │ Master  │ ←── 写入               │
  │  └────┬────┘                        │
  │       │                             │
  │  ┌────┴────┐                        │
  │  │         │                        │
  │  ▼         ▼                        │
  │ ┌──────┐ ┌──────┐                   │
  │ │Slave1│ │Slave2│ ←── 读取          │
  │ └──────┘ └──────┘                   │
  └─────────────────────────────────────┘

  Sentinel 职责：
  1. 监控：定期 PING 主从节点，检测是否存活
  2. 故障转移：Master 不可达时，投票选举新 Master
  3. 通知：故障转移后通知客户端新 Master 地址
```

**故障转移流程**：
1. Sentinel 发现 Master 不可达（主观下线 SDOWN）
2. 多数 Sentinel 确认不可达（客观下线 ODOWN）
3. 选举 Leader Sentinel 负责故障转移
4. Leader 从 Slave 中选择新的 Master（优先级 > offset > runid）
5. 将其他 Slave 指向新 Master
6. 通知客户端

```conf
# sentinel.conf
sentinel monitor mymaster 127.0.0.1 6379 2    # 至少2个哨兵同意才故障转移
sentinel down-after-milliseconds mymaster 5000 # 5秒不可达判定为主观下线
sentinel failover-timeout mymaster 60000       # 故障转移超时时间
```

### 6.3 Redis Cluster（官方集群方案）

Redis Cluster 是 Redis 3.0 引入的官方分布式方案，将数据分片、故障转移、节点发现全部集成在 Redis 自身中，无需外部组件（如哨兵）。它是目前大规模 Redis 部署的首选方案，支持最多 1000 个节点。

Redis Cluster 的核心设计理念是**去中心化**：每个节点都保存完整的集群拓扑信息，节点之间通过 Gossip 协议交换状态，客户端可以连接任意节点发起请求。如果请求的 key 不在当前节点上，节点会返回 MOVED 重定向，告知客户端正确的目标节点。

```
  Redis Cluster 架构
  ┌──────────────────────────────────────────────────────┐
  │                                                      │
  │   客户端                                              │
  │     │                                                │
  │     │ SET key1 "value1"                              │
  │     │                                                │
  │     ▼                                                │
  │   CRC16("key1") % 16384 = 5798                      │
  │     │                                                │
  │     │ 路由到负责 slot 5798 的节点                      │
```

## 七、Hash Slot 分片机制

### 7.1 16384 个槽

Redis Cluster 使用 16384 个哈希槽（Hash Slot）进行数据分片：

```bash
# 路由公式
slot = CRC16(key) % 16384

# 集群内部通信：每个节点通过心跳广播自己负责的槽位
# 槽位用 bitmap 表示：16384 位 = 2048 字节 = 2 KB

# 为什么是 16384 而不是 65536？
# 1. 心跳包中槽位 bitmap 约 2KB，节点数多时心跳开销可控
# 2. Redis Cluster 设计上限 ~1000 个节点
#    1000节点 × 2KB = 2MB/s 心跳带宽，可接受
# 3. 65536 槽位 bitmap 为 8KB，1000节点 = 8MB/s，开销过大
```

### 7.2 Hash Tag

通过 `{tag}` 语法强制相关 key 路由到同一个槽，支持跨 key 的原子操作：

```bash
# 以下两个 key 会被路由到同一个 slot
SET {user:1001}.name "Alice"
SET {user:1001}.age 28

# 计算 slot 时使用 {} 内的部分
CRC16("user:1001") % 16384   # 而非 CRC16("{user:1001}.name")

# Pipeline 跨 slot 问题
# 不使用 Hash Tag：不同 key 可能在不同节点，Pipeline 不可用
# 使用 Hash Tag：同 tag 的 key 在同一节点，Pipeline 可用
```

## 八、脑裂问题与防御

### 8.1 脑裂场景

```
  脑裂（Split Brain）
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  正常状态：                                       │
  │  Client ──→ Master(A) ──→ Slave(B)              │
  │                                                  │
  │  网络分区后：                                     │
  │  Client ──→ Master(A)  │  Master(A') ←── Slave  │
  │  (继续向旧Master写入)    │  (哨兵将B提升为新Master) │
  │                        │                          │
  │  问题：A 和 A' 都认为自己是 Master                 │
  │  客户端向 A 写入的数据在 A 恢复后会被丢弃           │
  │  数据丢失！                                       │
  └──────────────────────────────────────────────────┘
```

### 8.2 防御配置

```conf
# redis.conf（Master 节点配置）
min-replicas-to-write 1        # 至少有 1 个 Slave 同步才接受写入
min-replicas-max-lag 10        # 且 Slave 的复制延迟不超过 10 秒

# 脑裂发生时的效果：
# 旧 Master(A) 与新 Master(A') 网络不通
# A 无法收到任何 Slave 的 ACK
# min-replicas-to-write 条件不满足
# A 拒绝所有写入请求 → 避免数据丢失
```

## 九、CAP 定位

### 9.1 Redis Cluster 是 AP 系统

```
  CAP 定理分析
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  C（一致性）：Redis 使用异步复制                   │
  │    - Master 写入后立即返回客户端                    │
  │    - Slave 异步拉取数据                           │
  │    - Master 宕机时，未同步到 Slave 的数据丢失      │
  │    - 只能保证最终一致性                            │
  │                                                  │
  │  A（可用性）：                                     │
  │    - 少数节点故障时集群仍可服务                     │
  │    - 自动故障转移（秒级）                          │
  │    - 但负责该 slot 的 Master+Slave 全挂则不可用    │
  │                                                  │
  │  P（分区容错）：                                   │
  │    - 网络分区时多数派继续服务                       │
  │    - 少数派拒绝服务（保护数据）                     │
  │                                                  │
  │  结论：Redis Cluster 选择 AP                      │
  │  如果需要强一致性，应使用 ZooKeeper / etcd          │
  └──────────────────────────────────────────────────┘
```

## 十、内存管理

### 10.1 TTL 管理

```bash
# 设置过期时间
EXPIRE key 3600            # 秒
PEXPIRE key 3600000        # 毫秒
EXPIREAT key 1700000000    # Unix 时间戳（秒）

# 条件设置过期（Redis 7.0+）
SET key value EX 3600 NX   # 不存在时设置
SET key value EX 3600 XX   # 存在时设置
SET key value EX 3600 GT   # 新TTL > 当前TTL 时才设置
SET key value EX 3600 LT   # 新TTL < 当前TTL 时才设置
```

### 10.2 双清理机制

```
  Redis 过期键清理策略
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  1. 惰性删除（Lazy Expiration）                   │
  │     - 客户端访问 key 时检查是否过期                 │
  │     - 过期则删除并返回 nil                         │
  │     - 优点：CPU 友好（不主动扫描）                  │
  │     - 缺点：过期但未被访问的 key 持续占用内存       │
  │                                                  │
  │  2. 定期删除（Active Expiration）                  │
  │     - 每 100ms 执行一次                           │
  │     - 随机抽取 20 个带 TTL 的 key                 │
  │     - 删除其中已过期的                             │
  │     - 如果过期比例 > 25%，重复上述过程              │
  │     - 单次执行时间上限 25ms（避免阻塞主线程）       │
  │                                                  │
  │  两者配合：                                       │
  │  惰性删除保证访问时的正确性                        │
  │  定期删除防止内存泄漏                             │
  └──────────────────────────────────────────────────┘
```

### 10.3 大批过期危机

```
  问题场景：
  100万个 key 在同一秒过期
  → 定期删除来不及清理
  → 内存持续膨胀
  → 可能触发 OOM 淘汰策略

  解决方案：随机 TTL 偏移
  // Java 代码
  int baseTTL = 3600;
  int randomOffset = ThreadLocalRandom.current().nextInt(0, 300); // 0~5分钟随机偏移
  redisTemplate.expire(key, baseTTL + randomOffset, TimeUnit.SECONDS);

  // 效果：100万个 key 的过期时间分散在 3600~3900 秒之间
  // 避免集中过期导致的内存尖峰
```

### 10.4 内存淘汰策略

当内存达到 `maxmemory` 限制时，Redis 根据淘汰策略删除 key：

```conf
maxmemory 4gb
maxmemory-policy allkeys-lru    # 淘汰策略
```

```
  八大淘汰策略
  ┌────────────────────────────────────────────────────────┐
  │ 策略                 说明               适用场景         │
  │ ────                 ────               ────────        │
  │ noeviction           不淘汰,写入报错     不允许丢数据     │
  │                                                        │
  │ allkeys-lru          所有key中淘汰      通用缓存         │
  │                      最近最少使用的                      │
  │                                                        │
  │ volatile-lru         带TTL的key中       部分key需持久化  │
  │                      淘汰最近最少使用的                   │
  │                                                        │
  │ allkeys-lfu          所有key中淘汰      热点数据缓存     │
  │                      最不经常使用的                      │
  │                                                        │
  │ volatile-lfu         带TTL的key中       混合场景         │
  │                      淘汰最不经常使用的                   │
  │                                                        │
  │ allkeys-random       所有key中随机淘汰   轮询访问场景     │
  │                                                        │
  │ volatile-random      带TTL的key中随机   特殊场景         │
  │                                                        │
  │ volatile-ttl         带TTL的key中       尽快清理临时数据  │
  │                      淘汰剩余TTL最短的                   │
  └────────────────────────────────────────────────────────┘

  LRU vs LFU 的区别：
  LRU (Least Recently Used)：淘汰最久没被访问的
    → 偶发全量扫描会导致大量key被淘汰（冷数据污染）

  LFU (Least Frequently Used)：淘汰访问频率最低的
    → 更精准地保留热点数据
    → Redis 4.0+ 使用 8位 Morris 计数器近似实现
    → 推荐用于缓存场景
```

**Redis 的 LRU 并非精确 LRU**，而是近似 LRU：随机采样 N 个 key（默认 5），淘汰其中最久未访问的。采样数越大越接近精确 LRU，但 CPU 开销也越大。

### 10.5 淘汰策略选型建议

在实际生产环境中，淘汰策略的选择直接影响缓存命中率和系统稳定性。以下是基于不同业务场景的推荐策略：

对于**通用缓存场景**（如商品详情、用户资料缓存），推荐使用 `allkeys-lfu`。LFU 能够精准识别真正的热点数据，不会因为偶发的全量遍历（如后台任务扫描所有 key）而将热点数据误淘汰。这是 Redis 4.0 之后最推荐的缓存淘汰策略。

对于**混合场景**（部分 key 需要持久化，部分 key 是纯缓存），推荐使用 `volatile-lru` 或 `volatile-lfu`。将需要持久化的 key 不设置 TTL，纯缓存的 key 设置 TTL，Redis 只在带 TTL 的 key 中进行淘汰，保证持久化数据不被误删。

对于**不允许丢数据的场景**（如 Redis 作为分布式锁存储、消息队列），必须使用 `noeviction`。当内存满时拒绝写入，由应用层处理降级逻辑，避免因淘汰策略误删关键数据。

一个常见的生产事故是：使用 `allkeys-lru` 淘汰策略，结果 Redis 把分布式锁的 key 淘汰了，导致多个客户端同时获取到锁。因此，对于存储锁、会话等关键数据的 Redis 实例，务必使用 `noeviction` 策略，并确保内存充足。

```conf
maxmemory-samples 10    # 采样数，默认5，生产建议5~10
```

## 十一、键命名规范与 Value 优化

### 11.1 键命名规范

```
  推荐格式：{业务模块}:{数据类型}:{业务标识}

  好的示例：
  user:profile:1001          # 用户资料
  order:detail:20240101001   # 订单详情
  cart:items:1001            # 购物车
  rate:api:user:1001         # 限流计数
  session:token:abc123       # 会话

  避免的写法：
  user_1001_profile          # 分隔符不统一
  key1                       # 无业务含义
  very_long_key_name_xxx     # 过长浪费内存
```

### 11.2 Value 优化建议

```
  1. 控制 Value 大小
     - String 建议 < 10KB
     - Hash/List/Set/ZSet 元素建议 < 5000 个
     - 大 Value 会阻塞主线程（序列化/反序列化/网络传输）

  2. 拆分大 Key
     - 大 Hash 拆分为多个小 Hash
     - 大 List 按时间分片
     - 示例：user:friends:1001 → user:friends:1001:page:1, page:2

  3. 序列化选择
     - JSON：可读性好，但体积大
     - Protobuf/Kryo：体积小，但需要 Schema
     - 推荐：内部服务用 Protobuf，对外开放用 JSON

  4. 避免大 Key 操作
     - HGETALL → 改用 HSCAN 分页
     - SMEMBERS → 改用 SSCAN
     - KEYS * → 改用 SCAN
```

## 十二、面试题精选

**Q1：RDB 和 AOF 的区别是什么？生产环境怎么选？**

RDB 是某一时刻的全量快照，恢复快但可能丢失最近数据；AOF 记录每条写命令，数据更安全但文件更大、恢复更慢。生产环境推荐混合持久化（aof-use-rdb-preamble yes），兼具 RDB 的快速恢复和 AOF 的数据安全。如果追求极致恢复速度且能容忍分钟级数据丢失，可只用 RDB。

**Q2：Redis 为什么使用单线程模型？6.0 引入多线程后还是单线程吗？**

Redis 单线程是因为内存操作不是 CPU 瓶颈，IO 多路复用已能处理数万并发连接，且单线程避免了锁竞争和竞态条件的复杂性。6.0 引入的多线程仅用于网络数据的读取、协议解析和响应序列化，核心命令执行仍然是单线程串行的，因此仍然保证了命令级别的原子性。

**Q3：Redis Cluster 为什么使用 16384 个哈希槽？**

三个原因：（1）每个节点通过心跳广播自己负责的槽位，使用 bitmap 编码，16384 位 = 2KB。如果 65536 位 = 8KB，在 1000 节点集群中心跳带宽为 8MB/s，开销过大。（2）Redis Cluster 设计上限约 1000 个节点，16384 个槽足以均匀分配。（3）压缩比更好，2KB 的 bitmap 比 8KB 更容易压缩传输。

**Q4：什么是 Redis 脑裂？如何防御？**

脑裂是指网络分区导致旧 Master 和新 Master 同时存在，客户端可能向已失效的旧 Master 写入数据，这些数据在分区恢复后会被丢弃。防御方式：在 Master 上配置 `min-replicas-to-write 1` 和 `min-replicas-max-lag 10`，要求至少有 1 个 Slave 在线且复制延迟不超过 10 秒才接受写入。脑裂时旧 Master 无法满足条件，自动拒绝写入。

**Q5：Redis 过期键是如何清理的？如果大批 key 同时过期会怎样？**

Redis 使用惰性删除 + 定期删除双重策略。惰性删除在访问 key 时检查过期；定期删除每 100ms 随机采样 20 个带 TTL 的 key，删除已过期的，过期率 > 25% 则重复采样。大批 key 同时过期时，定期删除来不及清理，内存持续膨胀。解决方案：在设置 TTL 时加入随机偏移量（如基础 TTL + 0~300秒随机值），将过期时间分散开。

**Q6：allkeys-lru 和 allkeys-lfu 有什么区别？怎么选？**

LRU 淘汰最久未被访问的 key，偶发全量访问（如缓存预热遍历）会把热点数据误判为冷数据。LFU 淘汰访问频率最低的 key，更能反映真实的访问热度。Redis 4.0+ 推荐使用 `allkeys-lfu` 作为缓存淘汰策略。如果 Redis 版本低于 4.0，使用 `allkeys-lru`。

**Q7：fork 子进程做 RDB/AOF 重写时，内存会翻倍吗？**

不一定翻倍。fork 使用 COW（Copy-On-Write）机制，子进程与父进程共享物理内存页。只有父进程修改的内存页才会复制。如果期间写操作很少，额外内存消耗很小；如果写操作频繁（修改大量已有 key），额外内存可能接近翻倍。生产环境应监控 `used_memory_rss` 和 `mem_fragmentation_ratio`。

**Q8：Redis 的异步复制会导致数据丢失吗？有什么补救措施？**

是的，Redis 使用异步复制，Master 写入后立即返回客户端，不等待 Slave 确认。如果 Master 在数据同步到 Slave 之前宕机，这部分数据会丢失。补救措施：（1）开启 `min-replicas-to-write` 确保至少有 Slave 在线；（2）应用层实现幂等设计；（3）关键业务增加对账任务定期比对 Redis 和 DB 数据；（4）如需强一致性，使用 ZooKeeper/etcd 替代。
